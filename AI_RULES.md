# AI_RULES — Frontend Agent Workflow

## 1. 基本原则

1. **文档优先**：所有实现必须基于 `docs/` 目录下的文档体系，不得随意偏离已定义设计。
2. **最小可运行**：优先实现最小闭环，不追求一步到位。每次只完成一个可独立验证的功能。
3. **分层边界不可逾越**：前端层禁止直接调用 Agent 层或存储层，所有跨层通信必须经由 API 接口。
4. **可追溯**：每个代码改动需能回答「这个修改对应文档哪一条」。

---

## 2. 文档体系

| 文档                         | 用途                                               |
| ---------------------------- | -------------------------------------------------- |
| `docs/1-prd.md`              | 产品需求定义，禁止在实现中改变已定义的交互行为     |
| `docs/2-technical-design.md` | 技术实现方案，包含数据库设计、接口定义、模块拆解   |
| `docs/3-architecture.md`     | 系统架构蓝图，包含分层结构、数据流、执行链路       |
| `docs/4-development-plan.md` | 开发计划文档，包含功能模块、时间线、资源分配       |
| `docs/5-api-spec.md.md`      | API 接口规格文档，包含请求参数、响应格式、错误码等 |
| `docs/6-todo.md`             | 当前迭代任务清单，所有实现应在 todo 范围内         |

每次实现代码前，**必须确认**相关设计已在文档中定义。文档未覆盖的改动，需先补充文档再实现。

---

## 3. 代码生成约束

1. **TypeScript everywhere**：前端与后端均使用 TypeScript，禁止 `.js` 文件（`node_modules` 除外）。
2. **文件命名**：统一 kebab-case（如 `file-repo.ts`、`workflow-orchestrator.ts`）。
3. **禁止跨层直接调用**：
   - 前端 → 后端：经由 HTTP API 接口
   - Controller → Agent：经由 `Workflow Orchestrator`，不直接在 Controller 中实例化 Agent
   - Agent → 存储：经由 Repository，不在 Agent 内直接操作数据库
4. **单模块单一职责**：一个文件只做一件事，文件名应反映其职责。
5. **必须写注释的场景**：
   - 非显而易见的业务逻辑
   - 与文档设计不一致时的临时处理
   - 尚未实现的占位逻辑（`// TODO: ...`）

---

## 4. 模块职责边界

### 4.1 后端分层边界

```
Routes → Controller → Service → Workflow/Agent/RAG → Repository → DB
```

| 层级                   | 禁止做什么                       |
| ---------------------- | -------------------------------- |
| Routes                 | 不写业务逻辑，只做参数校验和分发 |
| Controller             | 不直接操作数据库，不实例化 Agent |
| Service                | 不写 SQL，不调用 LLM             |
| Workflow / Agent / RAG | 不做 HTTP 响应封装               |
| Repository             | 不写业务逻辑，只做 CRUD          |
| DB                     | 无业务逻辑                       |

### 4.2 前端分层边界

```
Pages → Components → Stores → API Services → HTTP
```

| 层级         | 禁止做什么                               |
| ------------ | ---------------------------------------- |
| Pages        | 不直接调用 LLM 或后端存储层              |
| Components   | 不维护跨组件状态，状态放 Store           |
| Stores       | 不直接发起 HTTP 请求，由 Services 层封装 |
| API Services | 只做 HTTP 调用，不做业务逻辑             |

### 4.3 Agent 边界

- 每个 Agent 是**纯函数**：相同输入必须产生相同结构输出
- Agent 内部禁止写死业务逻辑，所有判断依赖 Prompt 和输入数据
- Agent 输出必须符合 `AgentResult<T>` 统一包装：`{ success, data, summary, warnings }`
- Agent 之间只能通过 `WorkflowContext` 结构化对象传递数据，禁止共享变量或全局状态

---

## 5. Mock 优先原则

所有模块必须**先实现 Mock 版本**，验证流程打通后再替换为真实实现。

| 模块         | Mock 要求                                        | 验收条件                       |
| ------------ | ------------------------------------------------ | ------------------------------ |
| Agent        | 返回预定义的 TypeScript 对象，结构与真实输出一致 | 数据流能跑通，前端能展示       |
| RAG Searcher | 返回固定测试 Chunk 列表                          | Retriever 能拿到上下文         |
| LLM Client   | 直接返回 Mock 数据，接口与真实 Client 一致       | 可切换为真实 Client 不改调用方 |
| API 接口     | 已有后端实现时用真实接口，无后端时用 MSW 拦截    | 前端不感知 Mock 存在           |

Mock 数据必须**结构化**（TypeScript 类型定义），不得用 `console.log` 或随机字符串替代。

---

## 6. Workflow 设计约束

1. **执行顺序不可写死**：节点顺序由 `workflow/constants.ts` 中的枚举定义，不允许在代码中硬编码。
2. **节点必须可独立测试**：每个 Agent 必须能独立运行并返回结果，不依赖前置节点的实际执行。
3. **节点状态必须落库**：`workflow_nodes` 表记录每个节点的 `input_json`、`output_json`、`status`、`duration_ms`，不允许跳过。
4. **错误不阻断已执行节点**：单节点失败时，已执行节点的记录和结果保留，只标记当前失败节点和任务失败状态。

---

## 7. 迭代流程

每次代码实现前，必须输出：

```
1. 本次修改/新增的文件列表
2. 每个文件修改的理由（对应文档哪一条）
3. 如何运行验证
4. 如何确认功能正确
```

完成代码后，**按上述四点输出总结**，不得跳过。

---

## 8. 禁止行为

1. 不实现文档体系未定义的功能。
2. 不修改 `docs/` 目录外的文件而不说明理由。
3. 不一次性生成整个系统，每次只实现一个模块。
4. 不引入 `docs/2-technical-design.md` 之外的依赖（需先更新文档并说明）。
5. 不在 Agent 中写死业务判断逻辑。
6. 不跳过节点状态落库。
7. 不在前端直接调用后端业务逻辑层。
8. 不重构项目整体结构（如需重构，先出方案文档，经确认后执行）。
