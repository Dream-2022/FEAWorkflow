## 2. 项目背景

在前端工程开发中，开发者在面对组件增强、需求改造、影响分析、文档补全等任务时，通常需要手动完成以下流程：

1. 阅读 README、设计文档、目录结构和源码。
2. 找出与任务相关的组件、依赖文件和业务上下文。
3. 判断改动影响范围与潜在风险。
4. 输出实现方案、改动建议、文档说明或 PR 描述。

这个流程存在几个明显问题：

* **代码库理解成本高**：对于新项目或复杂项目，上手慢，依赖人工搜读。
* **分析链路不可视**：通用聊天式 AI 工具往往只给一段结果，缺少任务拆解、检索、推理、自检过程。
* **工程场景适配弱**：单轮问答难以真正结合项目上下文，输出容易“像答案，不像方案”。
* **结果不够结构化**：工程任务更需要“涉及文件、影响范围、建议方案、风险点、文档草稿”这类可执行结果，而不是泛泛叙述。

因此，需要一个面向前端工程场景的多 Agent 工作流平台，用于将“工程任务理解 → 上下文检索 → 实现分析 → 方案生成 → 自检审核”形成闭环，并以结构化方式展示结果与过程。

---

## 3. 项目目标

### 3.1 核心目标

构建一个面向前端工程任务的多 Agent 工作流平台，支持上传项目代码与文档后，自动完成任务规划、上下文检索、影响分析、方案生成、自检审核，并输出结构化改动建议与文档草稿。

### 3.2 MVP 目标

在 1 个月内完成一个可演示、可讲解、可写入简历的 Web 端版本，实现以下最小闭环：

**上传文档 / 代码 → 输入前端工程任务 → Planner 拆解 → Retriever 检索 → Analyzer 分析 → Generator 生成 → Reviewer 自检 → 输出结构化结果**

### 3.3 商业/产品价值

* 降低前端项目理解成本。
* 缩短组件增强、需求分析、文档输出的准备时间。
* 提升 AI 在工程场景中的可解释性和可落地性。
* 形成“不是聊天壳，而是工程工作流产品”的项目价值。

## 4. 用户与使用场景

### 4.1 目标用户

#### 用户类型 A：校招 / 初中级前端开发者

特点：

* 需要快速理解项目代码库。
* 面对组件改造、文档补全任务时容易无从下手。
* 希望 AI 给出结构化分析而不是泛泛回答。

#### 用户类型 B：组件库 / 中后台前端开发者

特点：

* 经常处理组件增强、API 变更、影响分析、文档更新。
* 需要快速定位相关文件和依赖。
* 对结果的可执行性和风险提示敏感。

#### 用户类型 C：个人开源项目维护者

特点：

* 希望快速生成 README、组件说明、变更描述。
* 需要更低成本维护文档和需求分析过程。

### 4.2 核心使用场景

#### 场景 1：组件增强分析

用户输入：为 Switch 组件新增 loading 与 disabled 联动能力，分析涉及文件并生成改动建议与 README 更新说明。

系统输出：

* 涉及文件列表
* 当前组件逻辑分析
* 推荐改动方案
* 风险点
* README 更新草稿

#### 场景 2：文档生成

用户输入：根据 Button 组件源码与目录，生成使用说明和 API 文档草稿。

系统输出：

* 组件用途总结
* Props / 事件 / 插槽说明
* 示例建议
* 文档草稿

#### 场景 3：需求影响分析

用户输入：将 message 组件从命令式调用改为支持队列管理，分析影响范围和改造方案。

系统输出：

* 影响模块
* 依赖文件
* 风险说明
* 实施步骤
* PR 描述草稿

## 6. 产品定位与设计原则

### 6.1 产品定位

这是一个**面向前端工程任务的 AI Agent Workflow 平台**，不是一个通用聊天机器人，也不是一个直接改代码的 IDE 插件。

### 6.2 设计原则

1. **可解释**：必须展示每个 Agent 的输入、输出、状态和耗时。
2. **工程化**：输出结果应服务于真实工程任务，而不是泛泛对话。
3. **结构化**：结果必须拆分为涉及文件、分析、方案、风险、文档草稿等模块。
4. **轻量闭环**：优先打通核心链路，而不是堆叠复杂技术。
5. **前端优势明显**：界面交互、可视化、结果组织是项目亮点之一。

## 7. 核心功能需求

## 7.1 文件上传与项目管理

### 功能说明

支持用户上传与当前工程任务相关的文档和源码文件，建立项目上下文。

### 支持类型

* Markdown 文档
* README 文件
* TS / JS 文件
* Vue / JSX / TSX 文件
* 组件目录（首版可支持多文件上传模拟目录）

### 输入

* 用户手动上传文件

### 输出

* 文件列表
* 文件基础信息（文件名、类型、大小、上传时间）
* 解析状态

### 规则

* 上传后自动触发解析与切片。
* 若解析失败，应提示失败文件与原因。
* 重复文件名可允许存在，但需通过唯一 ID 区分。

### 验收标准

* 用户可成功上传至少 20 个文件。
* 系统可展示上传列表与解析状态。
* 支持删除已上传文件并同步移除索引。

---

## 7.2 任务输入

### 功能说明

用户输入工程任务，作为 Agent Workflow 的起点。

### 支持任务类型

* 新增组件功能
* 分析模块改动影响范围
* 生成文档草稿
* 生成 README 更新说明
* 输出改造方案或 PR 描述

### 输入形式

* 自由文本输入
* 任务模板选择（第二阶段）

### 示例

* 为 Switch 组件新增 loading 与 disabled 联动能力。
* 根据 Button 组件源码生成 API 文档。
* 分析将 message 组件改为支持队列管理的影响范围。

### 验收标准

* 用户可输入任务并发起执行。
* 系统会记录任务文本、发起时间、执行状态。

---

## 7.3 多 Agent 工作流执行

### 功能说明

系统以多阶段 Agent 节点方式执行任务，而不是一次性调用模型返回结果。

### Agent 列表

#### 1）Planner Agent

职责：拆解任务，明确子任务和检索方向。

输入：

* 用户任务

输出：

* 子任务列表
* 推荐执行顺序
* 检索关键词
* 预估关注模块

#### 2）Retriever Agent

职责：根据任务和 Planner 输出，从已上传项目中检索最相关上下文。

输入：

* 用户任务
* Planner 输出
* RAG 索引

输出：

* 相关代码片段
* 相关文档片段
* 涉及文件路径列表
* 相关性说明

#### 3）Analyzer Agent

职责：分析当前实现、依赖关系、影响范围和潜在风险。

输入：

* 用户任务
* 检索结果

输出：

* 当前实现分析
* 影响范围
* 潜在依赖
* 风险点

#### 4）Generator Agent

职责：生成改动建议、文档草稿、PR 描述等工程结果。

输入：

* 用户任务
* Analyzer 输出

输出：

* 推荐改动方案
* 改动步骤
* 文档草稿
* PR 描述建议

#### 5）Reviewer Agent

职责：对生成结果做一致性检查和遗漏提示。

输入：

* Generator 输出
* 检索结果

输出：

* 不一致项
* 可能遗漏点
* 风险补充
* 最终校正版输出

### 状态流转

* idle
* planning
* retrieving
* analyzing
* generating
* reviewing
* done
* error

### 节点数据结构

每个节点需保存：

* nodeName
* status
* input
* output
* startTime
* endTime
* duration
* errorMessage

### 验收标准

* 完整工作流可按顺序执行。
* 任一节点执行状态可视化展示。
* 节点失败时可反馈错误信息。

---

## 7.4 RAG 检索模块

### 功能说明

对上传文档和代码进行切片、索引和检索，为 Agent 提供工程上下文。

### 切片规则

#### Markdown 切片

* 按标题层级切分。
* 按段落或小节切分。
* 每个 chunk 保留标题路径和来源文件路径。

#### 代码切片

* 按文件切分。
* 在可能情况下，按函数、组件、导出模块进一步切分。
* 每个 chunk 记录文件路径、代码类型、导出名称或组件名。

### 索引字段

* chunkId
* fileId
* filePath
* chunkType（doc/code）
* content
* summary（可选）
* keywords
* language
* createdAt

### 检索策略

MVP 建议支持两层：

1. 文件名/关键词匹配。
2. 内容相似度或 embedding 检索。

### 检索输出

* Top N 相关 chunk
* 来源文件
* 相关性得分或排序信息

### 验收标准

* 能基于任务关键词检索相关文件和片段。
* 检索结果可回显到界面。
* 检索结果可被后续 Agent 使用。

---

## 7.5 结构化结果展示

### 功能说明

系统最终输出应为面向工程执行的结构化结果，而不是一段长回复。

### 结果模块

1. 涉及文件列表
2. 当前实现分析
3. 推荐改动方案
4. 风险点
5. 文档草稿
6. 下一步建议

### 展示要求

* 每个模块单独成卡片。
* 涉及文件支持点击查看关联片段。
* 文档草稿支持复制。
* 结果需保留本次任务上下文。

### 验收标准

* 结果展示结构清晰，便于直接汇报或复制。
* 用户可一眼看到文件、分析、方案、风险和文档输出。

---

## 7.6 执行过程可视化

### 功能说明

将 Agent Workflow 的执行过程以可视化形式展示，降低“套壳感”，增强可解释性。

### 展示内容

* 节点顺序
* 当前执行状态
* 每个节点耗时
* 节点输入摘要
* 节点输出摘要

### 展示形式

* 流程节点图
* 时间线日志
* 状态标签（进行中/成功/失败）

### 验收标准

* 用户能明确看到当前执行到了哪个阶段。
* 可以展开查看某节点的输入输出摘要。

---

## 7.7 历史任务记录

### 功能说明

保存最近任务，便于用户回看和复用结果。

### 保存字段

* taskId
* 用户任务
* 执行时间
* 执行状态
* 最终结果摘要
* 涉及文件数量

### 交互要求

* 可查看最近任务列表。
* 可点击查看历史任务详情。
* 可删除历史任务记录。

### 验收标准

* 至少保留最近 20 条任务记录。
* 历史任务可重新查看结果页。

## 8.3 关键交互流程

### 流程 1：上传并执行任务

1. 用户上传文件。
2. 系统解析文件并建立索引。
3. 用户输入工程任务。
4. 点击“开始分析”。
5. 系统依次执行 Planner → Retriever → Analyzer → Generator → Reviewer。
6. 中间区域展示执行状态。
7. 右侧输出结构化结果。

### 流程 2：查看节点细节

1. 用户点击某个 Agent 节点。
2. 系统展示该节点输入、输出与耗时。
3. 用户可查看与节点关联的检索片段。

### 流程 3：查看历史任务

1. 用户点击左侧历史任务。
2. 系统加载任务详情与结果。
3. 用户可复用文档草稿或继续新任务。

## 9. 非功能需求

### 9.1 性能需求

* 单次任务平均响应时间控制在可接受范围内（目标 10~20s，视模型与文件量而定）。
* 任务执行期间界面必须实时反馈阶段状态。
* 上传和解析中不能导致页面明显卡死。

### 9.2 易用性需求

* 用户无需理解 Agent 原理，也能完成基本使用。
* 结果展示需直观、清晰、可复制。
* 空结果和错误场景要有明确提示。

### 9.3 稳定性需求

* 某个 Agent 节点失败时，不应导致整个页面崩溃。
* 历史任务和上传文件应能稳定保存。

### 9.4 可扩展性需求

* Agent 节点应支持继续扩展。
* 模型调用适配器应支持后续切换不同兼容接口。
* RAG 检索模块后续可升级为向量库方案。

## 13. 错误与异常场景

### 场景 1：未上传文件直接执行任务

处理方式：

* 提示用户先上传文件。
* 若允许纯任务执行，则显示“当前任务将基于空上下文执行，结果可能不准确”。

### 场景 2：文件解析失败

处理方式：

* 标记失败文件。
* 展示失败原因。
* 允许继续处理其余文件。

### 场景 3：检索结果为空

处理方式：

* 提示“未找到高相关上下文”。
* Generator 可基于任务文本做保守输出，并明确说明依据不足。

### 场景 4：模型调用失败

处理方式：

* 显示失败节点。
* 提供错误信息。
* 保留已完成节点结果。

### 场景 5：输出内容不完整

处理方式：

* 由 Reviewer 节点补充提示。
* 明确“以下内容为初步建议，建议人工确认”。

## 14. 验收标准

## 14.1 产品验收

* 支持上传文档与源码文件。
* 支持输入工程任务并执行多 Agent Workflow。
* 可展示 Planner、Retriever、Analyzer、Generator、Reviewer 五个节点。
* 可基于上传内容完成基本检索。
* 可输出结构化结果卡片。
* 可查看历史任务。
* 可看到执行状态和节点耗时。

## 14.2 演示验收

至少支持以下 2~3 个固定场景演示：

1. Switch 组件增强分析。

## 18. 一句话总结

Frontend Agent Workflow 是一个面向前端工程任务的多 Agent 工作流平台，通过“任务拆解 + 工程上下文检索 + 影响分析 + 方案生成 + 自检审核”的方式，把原本依赖人工阅读代码库和整理文档的流程，沉淀为一个可解释、可视化、可复用的产品闭环。
